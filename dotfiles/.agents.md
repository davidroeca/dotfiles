# Principles

## Avoid assumptions and present options to user

- If uncertain, ask rather than guess.
- Present multiple interpretations; don't just pick one silently.
- If a simpler approach exists, say so; push back when warranted.
- Stop when confused. Name what's unclear and ask for clarification.

## Aim for simplicity

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If 200 lines could be 50, rewrite it.

## Limit scope of changes

- Touch only the code that you must.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it but don't delete it.

## Goal-driven execution

Define success criteria prior to execution, leveraging TDD.

Examples:

- Write tests for invalid inputs, then make them pass.
- Write a test that reproduces a bug, then make it pass.
- Ensure tests pass before and after a refactor.

# Workflows

## Branching

Use `wt` (worktrunk) by default for all branch and worktree management:

```bash
wt switch --create <branch>  # create worktree + branch, switch into it
wt list                       # show active worktrees
wt merge                      # merge current branch into trunk
wt remove                     # remove worktree after merge
# wt <subcommand> --help for details
```

If `wt` is unavailable: `git worktree add -b <branch> ../<branch> HEAD`. Last resort: `git checkout -b <branch>`.
